# PLAN-0011: MVPユーザビリティ改善（判定UI・開始トリガー・体格補正）

- **Date**: 2026-03-18
- **Owner**: @tujing
- **Scope**: FE
- **Status**: Draft

## 1. 背景 / 目的（Context & Goals）

- 課題: 現状のフォーム判定は棒人間の関節色変化が主で、利用者にとって「どこが良い/悪いか」が直感的に分かりにくい。
- 課題: 撮影開始までに準備時間が必要だが、開始トリガーが手動ボタンのみで実運用に合いにくい。
- 課題: 教師/生徒の体格差補正に使う比率が固定前提で、ユーザーごとの適応ができない。
- 課題: `ViewController` / `ScaledPoseHelper` 周辺に責務集中があり、保守性が低い。
- 目的: MVPとして、短期間で体感価値を上げる改善を優先順位付きで投入する。
- 目的: 既存機能の回帰リスクを抑えるため、各改善にフォールバック経路を持たせる。

## 2. 非目標（Non-Goals）

- 高精度な自然言語音声認識（キーワード認識）による開始制御。
- 姿勢評価アルゴリズムの全面刷新。
- 録画基盤や保存フォーマットの大規模変更。
- 全面リファクタリングを単一リリースで完了すること。

## 3. 影響範囲（Impact）

- UI:
  - `PoseImageView` の判定可視化（色以外の表示）
  - 撮影開始導線（カウントダウン/自動開始トグル）
  - 体格補正設定UI（初回キャリブレーション）
- ロジック:
  - 開始トリガー判定（音量閾値または簡易イベント検知）
  - `TeacherStudentRatio` の永続化・読み出し
- 仕様/運用:
  - `docs/specs/ui.md` / `docs/specs/structure.md` / `docs/specs/tech.md` の更新
  - 設定未完了時のデフォルト挙動と復旧手順の明文化

## 4. 代替案とトレードオフ（Alternatives & Trade-offs）

- 代替案A: 先に全面リファクタリング（要求4）を実施してから機能追加。
  - 利点: 以降の実装基盤が綺麗になる。
  - 欠点: ユーザー価値の反映が遅く、MVPリードタイムが長い。
  - 却下理由: まず体験改善を出す方が投資対効果が高い。
- 代替案B: 音声開始を最初に実装。
  - 利点: 体感上の新規性が高い。
  - 欠点: 判定UIが分かりづらいままだと継続利用価値が上がりにくい。
  - 却下理由: 判定理解の改善（要求2）のほうが基礎価値として先。
- 選定理由:
  - 実装順を **要求2 → 要求1 → 要求3 → 要求4** とし、毎段でフォールバックを残す。

## 5. 実装方針（Design Overview）

- フェーズ1（要求2）: 判定UI改善
  - 関節色のみ表示から、3段階ラベル（OK/注意/NG）とズレ強調表示を追加。
  - 新表示が問題を起こした場合は旧色表示のみへ切り戻せるフラグ境界を用意。
- フェーズ2（要求1）: 開始トリガー改善
  - まず `10秒カウントダウン + 手動開始` を標準化。
  - 次段で簡易音量閾値トリガーを追加し、検知失敗時は手動開始へフォールバック。
  - 状態遷移は `idle -> countdown -> recording` を基本とし、任意タイミングで `manualStart` を許可する。
  - 音量トリガーは `設定ON` かつ `countdown` 中のみ評価し、閾値到達時は残りカウントをスキップして即時開始する。
  - 一定時間（例: 8秒）で閾値未到達なら自動開始を中止し、手動開始ボタン強調表示へ切替える。
  - カウントダウン中の表示要件は「残り秒数」「自動開始ON/OFF状態」「手動開始ボタンの常時表示」を必須とする。
- フェーズ3（要求3）: 体格補正設定
  - 初回キャリブレーション画面を追加し、`TeacherStudentRatio` を保存/読込。
  - 未設定・破損時は既存デフォルト比率へフォールバック。
- フェーズ4（要求4）: 重点リファクタ
  - 先行フェーズで触れた範囲に限定して責務を分離し、`ViewController` の肥大化を抑える。

## 6. 実装手順（Implementation Steps）

1. 判定UI改善（優先度P1）
   - `PoseImageView` の可視化ロジックを拡張し、3段階評価ラベルと要注意関節の明示を追加。
   - 旧表示を維持した切替境界を用意し、回帰時の切り戻しを可能にする。
2. 撮影開始トリガー改善（優先度P1）
  - 録画開始前にカウントダウンUIを導入。
  - 手動開始を常に残したうえで、簡易音量閾値トリガーをオプション追加。
  - 自動開始ON時は音量メータ値を評価し、閾値到達で即時録画開始、未到達タイムアウト時は手動へ誘導する。
  - 状態は `idle -> countdown -> recording` を維持し、複雑な状態追加は行わない。
  - 手動開始/カウントダウン満了/音量閾値到達を単一の録画開始入口へ集約し、先頭で二重発火をガードする。
3. 体格補正設定（優先度P2）
   - 初回または設定画面から比率を登録/更新できるUIを追加。
   - 保存層（例: `UserDefaults` ラッパ）を作成し、`TeacherStudentRatio` 初期化時に適用。
4. 最小リファクタ（優先度P3）
   - 新規追加ロジックを `ViewController` 直書きせず、小さな責務単位に分離。
   - `ScaledPoseHelper` の重複処理を段階的に共通化し、可読性を改善。

## 7. テスト計画 / 受入基準（Test Plan / Acceptance Criteria）

- 単体テスト:
  - 判定UI用スコア閾値ロジックの境界値（OK/注意/NG）
  - 開始トリガー状態遷移（countdown完了/手動開始/音量閾値到達/タイムアウト）
  - 比率保存/読込の正常系・未設定時フォールバック
- 結合確認:
  - 撮影開始フロー（手動/カウントダウン/音量トリガー有効時）
  - 録画開始イベントが重複発火しないこと（手動連打/閾値到達同時発生）
  - 既存録画保存・セッション再生への回帰がないこと
- 手動確認:
  - 判定表示が従来より理解しやすいこと（関節単位の改善点が把握できる）
- DoD:

  - [ ] 音量トリガーをオプションで実装（失敗時手動開始へフォールバック）
  - [ ] 開始トリガー状態遷移の単体テストを実装
  - [ ] 録画開始の重複発火防止を確認

## 8. ロールバック / リスク / 監視（Rollback, Risks, Monitoring）

- ロールバック:
  - 判定UI/開始トリガー/比率設定は個別に無効化できる境界を持たせる。
- リスク:
  - 音量トリガーは環境ノイズに影響されやすく、誤検知/未検知の可能性がある。
  - 比率設定UIが煩雑だと離脱要因になる。
  - 自動開始と手動開始が競合した場合に二重開始する可能性がある。
- 軽減策:
  - 既定は手動開始を残し、自動開始はオプション扱いにする。
  - 開始トリガーは録画開始後に即無効化し、二重発火をガードする。
  - タイムアウト時は明示的に「手動開始へ切替」のUI表示を出す。
  - 比率未設定時は固定比率で動作継続し、設定を後回しにできる導線にする。

## 11. 次回実装スコープ（開始トリガー: 自動開始トグル）

- 対象:
  - 既存の10秒カウントダウン表示と手動開始導線を維持
  - 自動開始ON/OFFトグルのUI追加
  - 音量閾値スライダーのUI追加
  - ON/OFFと閾値の `UserDefaults` 永続化
  - countdown中のみ音量監視を有効化
  - 閾値到達時の即時開始、未到達タイムアウト時の手動誘導
  - 手動開始/カウントダウン満了/閾値到達の録画開始入口を1か所に統一
- 非対象:
  - 体格補正UI・保存
  - 状態管理の大きな抽象化
  - 開始トリガーの全面リファクタ
  - 複雑な状態機械化（`awaitingManual` などの新状態追加）
  - 音声認識の高精度化
- 受入基準:
  - 自動開始をOFFにした場合、countdown満了または手動開始のみで録画開始する。
  - 自動開始をONにした場合、countdown中に閾値到達で即時開始する。
  - 自動開始の設定値（ON/OFF、閾値）がアプリ再起動後も保持される。
  - タイムアウト時に手動開始へ誘導し、録画開始が二重発火しない。
