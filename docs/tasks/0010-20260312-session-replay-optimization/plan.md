# PLAN-0010: セッション再生最適化

- **Date**: 2026-03-12
- **Owner**: @tujing
- **Scope**: FE
- **Status**: Approved

## 1. 背景 / 目的（Context & Goals）

- 課題: `SessionDetailView` の Pose 再生は `pose.ndjson` を全件読み込んで `PoseFrame` 配列を構築しており、長尺または高fpsセッションで読み込み待ちとメモリ使用量の増加が起きやすい。
- 課題: `docs/specs/tech.md` にも「長尺/高fpsセッション向けのメモリ最適化」が既知課題として残っている。
- 目的: 長尺セッションでも詳細画面を安定表示できるよう、Pose フレーム読み込みと再生同期の実装を軽量化する。
- 目的: MVP の既存体験を壊さず、再生開始までの待ち時間とメモリ負荷を下げる。
- 定量目標: iPhone 15 Simulator（iOS 17.0.1）で 10 分相当セッションを用いた計測において、初回表示時間とピークメモリ増分を現行比で 30% 以上改善する。

## 2. 非目標（Non-Goals）

- フォーム採点や比較 UI の追加。
- クラウド同期や DB 導入。
- 録画パイプライン自体の刷新。
- 再生 UI の全面 redesign。

## 3. 影響範囲（Impact）

- UI: `SessionDetailView` / `PosePreviewView` の読み込み状態と再生同期挙動に影響。
- 永続化: `pose.ndjson` の保存形式は維持し、読み込み方式のみ変更候補。
- 性能/可用性: メモリ使用量、初期表示速度、長尺再生時の安定性に直接影響。
- テスト: Repository / ViewModel / 再生同期ロジックの単体テスト追加が必要。
- リリース/運用: 長尺セッションの動作確認手順を明文化する。

## 4. 代替案とトレードオフ（Alternatives & Trade-offs）

- 代替案A: 現行の全件読み込みを維持し、端末性能に依存させる。
  - 利点: 実装変更が最小。
  - 欠点: セッション長に比例してメモリ負荷が増え、既知課題が解消しない。
  - 却下理由: MVP 完了後の次優先課題として妥当でない。
- 代替案B: `pose.ndjson` を別バイナリ形式へ変換して保存し直す。
  - 利点: 読み込み効率を大きく改善できる余地がある。
  - 欠点: 保存フォーマット変更、互換性対応、検証コストが大きい。
  - 却下理由: 現時点ではスコープが広すぎる。
- 選定理由: 保存形式は維持しつつ、インデックス化・遅延読み込み・キャッシュ制御で再生経路を軽量化する案が、コストと効果のバランスが良い。

## 5. 実装方針（Design Overview）

- `RecordingSessionRepository` または専用ローダー層で `pose.ndjson` の走査方法を見直し、全件常駐を避ける。
- 再生時刻から必要フレームへ到達しやすいよう、最低限の時刻インデックスを構築する。
- `SessionDetailViewModel` は再生中に必要な前後フレームのみ保持し、UI には現在フレームだけを渡す。
- 長尺セッション用のテストデータを用意し、読み込み完了時間とメモリ増加の悪化を防げているか確認する。

## 6. 実装手順（Implementation Steps）

1. 現行調査
   - `SessionDetailViewModel` / `RecordingSessionRepository` / Pose 再生ロジックの読み込み経路を整理する。
   - 長尺セッションでのボトルネックを特定し、改善対象をコード上で明確化する。
   - ベースライン計測（初回表示時間/ピークメモリ増分）を取得し、`impl-report.md` に記録する。
2. 読み込み戦略の変更
   - `pose.ndjson` 全件配列化を避けるローダーへ置き換える。
   - 再生時刻から参照するフレーム取得 API を定義する。
   - 旧実装を切り戻し可能な形で分離し、差し替え境界を明確化する。
3. 再生同期の調整
   - ViewModel 側での再生ループを新ローダーに接続する。
   - UI のローディング表示やエラー表示が必要なら最小限追加する。
   - 同期ずれ時に旧取得経路へ戻せるフォールバック手段を用意する。
4. テストと仕様反映
   - 単体テストを追加し、既存挙動を維持できていることを確認する。
   - 改善後の再計測を行い、ベースラインとの差分を記録する。
   - `docs/specs/tech.md` / `docs/specs/structure.md` の該当箇所を更新する。

## 7. テスト計画 / 受入基準（Test Plan / Acceptance Criteria）

- 単体テスト:
  - 長尺相当の `pose.ndjson` 入力で必要フレームを正しく取得できること。
  - 先頭/末尾/中間時刻で期待どおりのフレームを返すこと。
- 結合確認:
  - セッション詳細画面で動画再生と Pose 同期オーバーレイが継続して表示されること。
  - 既存の短尺セッションで回帰がないこと。
- 性能計測:
  - 条件: iPhone 15 Simulator（iOS 17.0.1）、同一 10 分相当セッションデータ、3 回計測の中央値を採用。
  - 指標:
    - 詳細画面初回表示時間（画面表示開始〜Pose オーバーレイ初回描画）
    - 詳細画面表示中のピークメモリ増分（表示直前との差分）
  - 合格基準:
    - 初回表示時間: ベースライン比 30% 以上改善
    - ピークメモリ増分: ベースライン比 30% 以上改善
- DoD:
  - [ ] 長尺セッション向け読み込み最適化を実装
  - [ ] 自動テスト追加
  - [ ] ベースライン/改善後の計測結果を `impl-report.md` に記録
  - [ ] 初回表示時間・ピークメモリ増分ともに 30% 以上改善
  - [ ] `docs/specs/tech.md` / `docs/specs/structure.md` 更新
  - [ ] リリースノート更新

## 8. ロールバック / リスク / 監視（Rollback, Risks, Monitoring）

- ロールバック: 新ローダーで不具合が出た場合、現行の全件読み込み実装へ戻せるよう責務分離して実装する。
- ロールバック手順:
  - 旧ローダー実装を保持し、切替ポイントを `SessionDetailViewModel` 側に限定する。
  - 不具合時は切替を旧実装に戻し、`impl-report.md` の Amendments/Errata に記録する。
- リスク: ランダムアクセス効率を優先しすぎると実装が複雑化する。まずは読み込み量削減を優先し、保存フォーマット変更は避ける。
- リスク: 再生同期がずれる可能性があるため、既存短尺セッションでも回帰確認を行う。
- 監視: アプリ内監視は未導入のため、テストデータと手動再生確認を証跡化する。
