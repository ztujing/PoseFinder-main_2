# docs ディレクトリ運用ガイド

このディレクトリは、本プロジェクトの **仕様・設計・タスク計画/実装ログ** を集約する場所です。  
詳細な規約はリポジトリルートの `AGENTS.md` を参照してください。

---

## 1. プロジェクト準備フロー

新しいプロジェクト（または大きめの新規開発案件）を始めるときは、次の手順で環境とドキュメントを整えます。

1. **リポジトリ構成の準備**

   - プロジェクト用リポジトリルート（例: `/repo`）を作成する。
   - この `docs` ディレクトリを `/repo/docs` として配置する。

2. **AGENTS.md の配置**

   - ルートに運用規約ファイル `AGENTS.md` を配置する（既存プロジェクトに組み込む場合は統合する）。
   - テンプレートとして `_AGENTS.md` がある場合は、`/repo/AGENTS.md` に移動・リネームするか、既存の `AGENTS.md` と内容を統合する。

3. **概要設計（specs）の作成**

   - `docs/specs/product.md` に、プロダクトの目的・ターゲットユーザー・主要機能・ビジネス目標などを記載する。
   - `docs/specs/tech.md` に、使用する技術スタック・ツール・技術的制約・方針を記載する。
   - `docs/specs/structure.md` に、ディレクトリ構成・命名規則・アーキテクチャ上の方針を記載する。
   - これらは AI に壁打ちしながら整理してよい（例: 「product.md を一緒に埋めて」と指示）。

4. **開発基盤のセットアップ**
   - 概要設計がある程度固まったら、選定したフレームワークをインストールする（例: Next.js, バックエンドフレームワークなど）。
   - ローカル開発用の Docker 構成を作成/調整する（詳細は `AGENTS.md` やプロジェクトの README を参照）。

## 2. Issue ごとの実装フロー

1 つの Issue（またはタスク）を進めるときは、以下のフェーズで AI に指示しながら進めます。

1. **DRAFT**

   - `docs/tasks/draft.md` に、今回の Issue の概要ややりたいことをラフに記載する。
   - 場合によっては、この過程はスキップしAIとチャットした後すぐにPLANを作成する場合もある。
   - 仕様やややこしい場合だけ、draft.mdに記載する。

2. **PLAN**

   - AI へ「タスクフォルダを作成し、実装 PLAN を作って」と指示する。
   - 生成された `plan.md` の内容を読み、必要に応じて修正点を AI に指示する。

3. **TASKS**

   - AI へ「実装 PLAN から、実装タスクを作って」と指示する。
   - 粒度や順序が適切かを確認し、必要ならタスクの分割・統合を AI に依頼する。

4. **IMPL（実装）**

   - 問題がなければ、AI に「タスクに基づいて実装して」と指示し、`tasks.md` のチェックボックスを更新しながら進める。
   - 実装途中で仕様変更が発生した場合は、必要に応じて `plan.md` / `tasks.md` を更新する。

5. **REVIEW 準備**

   - 実装が完了したら、AI に「実装レポートを作成して」と指示し、`impl-report.md` を生成する。
   - `impl-report.md` のサマリ / 差分 / テスト結果が妥当かを確認し、必要に応じて修正を依頼する。

6. **AI REVIEW**

   - AI に「実装をレビューして」と指示し、コードと `impl-report.md` の内容にギャップがないか、自動レビューを受ける。
   - 指摘事項があれば、次の FIX フェーズで対応する。

7. **FIX（修正）**

   - AI の指摘や自己レビューで見つかった問題を修正するよう指示する。
   - 修正に応じて `impl-report.md` や `tasks.md` の状態も更新する。

8. **HUMAN REVIEW（人間によるレビュー）**
   - 最終的に、実装者以外の人が `impl-report.md` に基づいてレビューする。
   - 問題がなければ、コミットして PR を作成し、Issue / Plan / Impl / ADR / Specs へのリンクを PR に記載する。

## 3. ADR の扱い

ADR（Architecture Decision Record）は、次のような場合に作成します。

- これまでの方針から **大きく変更** する必要があるとき。
- 複数の選択肢の中から 1 つを選び、**ミーティング等での決定理由を将来に残したい** とき。

作成ルールの概要:

- `docs/adr/####-YYYY-MM-DD-<slug>.md` の形式でファイルを作成する。
- 1 つの ADR には原則として **1 つの設計判断** を記録する。
- 新しい ADR が既存の ADR を置き換える場合は、
  - 新 ADR に **Supersedes: ADR-XXXX** を明記し、
  - 旧 ADR の Status を **“Superseded by ADR-XXXX”** に更新する（本文は変更しない）。

## 4. 参考: 主なドキュメント

- `AGENTS.md`  
  本リポジトリ全体の運用規約（SSOT）。docs 配下の詳細ルールもここに定義。

- `docs/specs/product.md`  
  プロダクトの目的・ターゲット・主要機能・ビジネス目標。

- `docs/specs/tech.md`  
  技術スタック・開発ツール・技術方針。

- `docs/specs/structure.md`  
  ディレクトリ構成・命名規則・アーキテクチャ方針。

- `docs/tasks/draft.md`  
  次に着手するタスクの仕様ドラフトをラフに書くための共通メモ。

以降の詳細な運用や更新ルールは、必ず `AGENTS.md` を優先して参照してください。
